🛑 Critical — Fix Immediately

C1 Homepage <title> and meta description leak from reviews page confirmed on 10 locale homepages

Observed: <title>What do people say about us? · Peach Bitcoin</title> and description "Do you want to earn 500 sats? Leave us a review in Apollo…" on /index.html. The full crawl also shows the same generic fallback title "Peach Bitcoin · Buy & Sell Bitcoin peer-to-peer" shared by 10 pages (homepages of several locales, T&C pages, and FAQ roots).

Root cause: the page template compiling to dist/index.html is rendering with the reviews-page title / description block.

Fix: src/includes/template.pug:4 falls back to siteData.meta.title + tagline if no page-level title is set. Audit the homepage .pug to remove the stale block, or set title/description explicitly to marketplace copy.

C2 Zero rel="canonical" sitewide 930 / 936 pages

Only 6 pages have a canonical — and those are the JS-redirect stubs like /how-it-works/ (which ironically use it correctly). Every real content page lacks it.

Impact: / and /index.html both return 200 with different HTML sizes. Same risk propagates to all 930 pages.

Fix: add to the head block in src/includes/template.pug:15-48:

link(rel="canonical" href=`https://peachbitcoin.com${canonicalPath || permalink}`)

C3 Zero hreflang tags sitewide 936 / 936 pages

Not a single page on the site emits hreflang. The sitemap already contains the localized counterparts (EN 205, DE 147, ES/FR/IT/PT 146 each), so the data is there — it just isn't being surfaced in the HTML.

Fix: emit a full cluster + x-default per page in src/includes/template.pug:

each l in ['en','es','de','it','fr','pt']
  link(rel="alternate" hreflang=l href=`https://peachbitcoin.com${getUrl(permalink, l)}`)
link(rel="alternate" hreflang="x-default" href=`https://peachbitcoin.com${getUrl(permalink, 'en')}`)

Handle edge case: some posts exist only in EN (e.g. 19-word blog stubs) — omit hreflang for locales where the translation doesn't exist, don't stub-link.

C4 Sitemap lists /index.html URLs instead of clean URLs 936 / 936 entries

Root cause: tasks/generate_sitemap.js:6-8 globs dist/**/*.html and string-replaces the path. Every URL in sitemap.xml ends in /index.html.

Fix:

const pages = html.map(f => f
  .replace(/.*\/dist/, 'https://peachbitcoin.com')
  .replace(/\/index\.html$/, '/'))

C5 Massive duplicate-title problem 480 / 936 pages affected (51%)

The full crawl found 28 duplicate-title groups covering 480 pages. Worst offenders:

  • "How to Buy Bitcoin on Peach Bitcoin Peer-to-Peer marketplace · Peach Bitcoin" — 55 pages (every programmatic /buy-bitcoin-with-*/ page shares this identical title)
  • "Buy & Sell Bitcoin using any payment method 2025 with PeachBitcoin · Peach B…" — 51 pages (every EN /faq/Buy-&-Sell-Bitcoin-using-[currency]-in-2024/ page)
  • Same translated pattern for PT, FR, IT, ES, DE — 44 pages each in every locale
  • "Blog · Peach Bitcoin" — 60 pages (blog index + tag pages across locales)
  • "Peach Blog · Peach Bitcoin" — 12 pages (EN blog + tag pages)
  • "Peach Bitcoin · Buy & Sell Bitcoin peer-to-peer" — 10 pages (multiple locale homepages)

Impact: duplicate titles are the single strongest cannibalization / doorway-page signal. The 55 programmatic /buy-bitcoin-with-*/ pages sharing an identical title is especially risky — Google's duplicate-content heuristics can demote or deindex the set.

Fix:

  • Programmatic pages: template titles like "Buy Bitcoin with {paymentMethod} — No KYC · Peach Bitcoin"
  • FAQ currency pages: "Buy & Sell Bitcoin with {currency.toUpperCase()} in 2025 · Peach Bitcoin"
  • Blog tag pages: "{Tag} — Peach Blog · Peach Bitcoin"
  • Locale homepages: distinct, localized, marketplace-centric titles

C6 312 URLs contain a literal & character 33% of the site

Paths like /faq/Buy-&-Sell-Bitcoin-using-vnd-in-2024/. Ampersand is a reserved URL character that must be percent-encoded as %26 — Google Search Console, social shares, copy/paste, analytics, and many crawlers all mishandle them. Several tools drop these URLs entirely.

Fix: rename the FAQ currency pages to replace & with and or just omit it (e.g. /faq/buy-sell-bitcoin-using-vnd-2025/). Requires slug changes in content/{en,es,de,it,fr,pt}/faq/ + Cloudflare 301s from the old paths.

H2 Missing <h1> on 24 indexable pages, duplicate <h1> on 24 more

Missing H1 (24 pages): these templates consistently lack an H1 across all 6 locales:

  • /kycfree-orderbook/ (×6 locales)
  • /how-to-buy-btc-no-kyc/ (×6 locales)
  • /referral/ (×6 locales)
  • /terms-and-conditions/ (×6 locales)

Multiple H1 (24 pages): all 6 locale homepages (desktop + mobile DOM dupes) + a handful of blog posts (peach-in-japanese, 1y-anniversary, Alternative-to-AgoraDesk × locales).

Fix: ensure exactly one <h1> per page. Collapse desktop/mobile into one element and show/hide with CSS.

H3 Unrendered markdown leaks into meta descriptions 37 pages

Literal **foo** asterisks appear in 37 pages' meta descriptions. Confirmed on PT homepage, multiple PT FAQs, EN T&C, PT blog posts. Not just a one-off ES bug — it's a sitewide content-pipeline issue.

Fix: either (a) author meta strings as plain text in the content files, or (b) strip markdown at build time before emitting the <meta name="description"> tag. A simple regex stripper (.replace(/\*\*(.+?)\*\*/g, '$1')) applied in the template render step would cover it.

H4 /how-it-works/ uses client-side JS redirect

Observed: <script>window.location.replace("/how-to-buy-btc-no-kyc/")</script> with noindex,nofollow.

But: the main nav in src/includes/template.pug:79 still links to /how-it-works/ via +navItem("how-it-works", ...), forcing an extra hop on every click.

Fix: point the nav directly at /how-to-buy-btc-no-kyc/, deploy a Cloudflare 301 Rule for the legacy path, then remove the stub HTML from the build.

H5 Empty og:site_name (sitewide)

src/includes/template.pug:24 uses content=siteData.title, but the string lives at siteData.meta.title — so every page emits <meta property="og:site_name"/> with no value.

Fix: change to content=siteData.meta.title or a literal "Peach Bitcoin".

H6 noindex pages still included in sitemap 6 pages

All 6 locale variants of /blog/group-hug/ emit <meta name="robots" content="noindex…"> but are still listed in sitemap.xml. Contradictory crawl signal — Google will log warnings and potentially distrust the sitemap.

Fix: either (a) remove the noindex if the posts should be indexed, or (b) exclude noindex pages from the sitemap output in tasks/generate_sitemap.js (fetch each built HTML and skip if noindex present).

H7 html lang mismatch on 5 localized pages

The PT, IT, FR, ES, and DE versions of /blog/group-hug/ emit <html lang="en"> despite living at localized URLs. These posts appear to be English-only content republished at localized paths (19 words each — essentially stubs).

Fix: either translate the post, remove from localized sitemaps/routes, or set proper lang attribute. Combined with H6, simplest solution: de-list them from the localized sitemaps.

H8 381 URLs have uppercase letters in their slug

Examples: /buy-btc-with-Amazon/, /faq/Buy-&-Sell-Bitcoin-using-vnd-in-2024/, /blog/Alternative-to-AgoraDesk/. Google treats URLs as case-sensitive on case-sensitive hosts (GitHub Pages is), so mixed-case inconsistency risks duplicate-indexation splits and makes URL sharing fragile.

Fix: normalize all slugs to lowercase kebab-case in content/**. Add Cloudflare 301s from old capitalized URLs to new lowercase ones.

Same pattern as C5 titles. Many FAQ and programmatic pages share identical descriptions.

M4 1,008 images missing alt attribute 930 of 936 pages

Out of 8,149 total <img> elements sitewide, 1,008 (12.4%) have no alt attribute. The problem is present on nearly every page, so it's a template / component issue rather than a per-page content miss.

Fix: (a) audit the partials / mixins that emit <img> in src/ and require an alt param; (b) decorative SVG icons should use alt="" + role="presentation" rather than omit alt; (c) add a build-time lint that fails on missing alt.

M6 Missing Content-Security-Policy header

What This Audit Did NOT Measure

Requires credentials / extra config:
  • Real Core Web Vitals (CrUX field data) — needs PAGESPEED_API_KEY for /seo google pagespeed + /seo google crux
  • Search Console data (clicks, impressions, index coverage, crawl errors) — needs OAuth via python scripts/google_auth.py

  • Full Lighthouse / rendered-DOM run — this crawl parses static HTML only. JS-rendered content and interaction metrics require Playwright-driven per-page capture via seo-performance and seo-visual subagents.
